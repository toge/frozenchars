# frozenchars 改善点・機能候補の総棚卸し

## 成果物

- コード全体の調査結果に基づく**全候補リスト**(バグ修正・API 拡充・テスト不足・ドキュメント・ビルド/CI・AGENTS.md 整合)をカテゴリ別に網羅し、優先度(高/中/低)を付記。
- 上位優先度項目については**具体的な実装ステップ**を記載。
- 新機能候補は**保守的・中規模・長期展望**の 3 層で提示。

> **注**: 調査過程で explore サブエージェントが `frozenchars_audit_report.md` を作業ツリーに生成してしまった(41KB)。計画モードのため削除不可。実装フェーズ冒頭で `rm frozenchars_audit_report.md` を実行すること。

## 優先度判定基準

- **高**: 既存機能のバグ・ビルド崩壊リスク・公開 API の重要な欠落・CI/パッケージ不整合
- **中**: テスト不足・ドキュメント欠落・保守性向上・中規模 API 拡充
- **低**: 長期展望の大規模機能・コミュニティファイル整備

### 実装 5: `FrozenString` 基本 API 拡充(C1, C2, C3, C4, C5)

**対象**: `include/frozenchars/string.hpp`

**問題**: 比較演算子、`operator[]`、`empty()`、イテレータ、`std::hash` が不在。`std::string_view` 互換 API の一部が欠落。

**手順**:
1. `string.hpp` の `FrozenString<N>` 定義に以下を追加(全て `constexpr`):
   - `[[nodiscard]] constexpr auto operator[](size_t i) const noexcept -> char` → `buffer[i]`(境界チェック無し、`std::string_view` と同様の UB 仕様)。必要なら `at(size_t)` も別途追加(境界チェック + `throw std::out_of_range`)。
   - `[[nodiscard]] constexpr auto empty() const noexcept -> bool` → `length == 0`。
   - `[[nodiscard]] constexpr auto begin() const noexcept` → `buffer.data()`。
   - `[[nodiscard]] constexpr auto end() const noexcept` → `buffer.data() + length`。
   - `[[nodiscard]] constexpr auto cbegin() const noexcept` → `begin()`。
   - `[[nodiscard]] constexpr auto cend() const noexcept` → `end()`。
   - `[[nodiscard]] constexpr auto front() const noexcept -> char` → `buffer[0]`(事前条件 `length > 0`)。
   - `[[nodiscard]] constexpr auto back() const noexcept -> char` → `buffer[length - 1]`(事前条件 `length > 0`)。
2. 比較演算子をフリー関数として追加(`string.hpp` 末尾、`operator+` の近く):
   - `operator==`, `operator<=>` を `FrozenString<M>` vs `FrozenString<N>`、`FrozenString<N>` vs `string_view`、`FrozenString<N>` vs `char const (&)[M]` の 3 系列について定義。`<=>` を使えば `<, <=, >, >=, ==, !=` が自動導出(C++20)。`string_view` への暗黙変換で衝突しないよう、`FrozenStringLike` 制約で SFINAE を入れる。
3. `std::hash<FrozenString<N>>` 部分特殊化を `string.hpp` 末尾(`namespace std` ブロック)に追加:
   ```cpp
   namespace std {
     template <size_t N>
     struct hash<frozenchars::FrozenString<N>> {
       size_t operator()(frozenchars::FrozenString<N> const& s) const noexcept {
         return hash<string_view>{}(s.sv());
       }
     };
   }
   ```
4. `test/test_simple.cpp` または新設 `test/test_frozen_string_api.cpp` に以下を追加:
   - `static_assert(FrozenString<1>{}.empty())`、`static_assert(!"abc"_fs.empty())`。
   - `static_assert("abc"_fs[1] == 'b')`。
   - `static_assert("abc"_fs.front() == 'a' && "abc"_fs.back() == 'c')`。
   - range-based for: `for (char c : "abc"_fs) {...}` がコンパイル可能なことを `STATIC_CHECK` または `REQUIRE` で検証。
   - `static_assert("abc"_fs == "abc"_fs)`、`static_assert("abc"_fs != "abd"_fs)`、`static_assert("abc"_fs < "abd"_fs)`、`"abc"_fs == std::string_view("abc")`、`"abc"_fs == "abc"`(リテラル)。
   - `std::unordered_map<FrozenString<4>, int>` を構築し `m["abc"_fs] = 1` ができることを `REQUIRE`。
5. 既存の暗黙 `string_view` 変換による比較と新演算子の曖昧さを確認。必要なら `string_view` への暗黙変換演算子を `explicit` に変更(ただし既存コードへの影響大なので慎重に。`to_sv()` が既にあるので `explicit` 化は推奨)。

**検証**: `bash build.sh && bash test.sh`。新テストケース全 pass。既存テストの回帰無し。`std::unordered_map<FrozenString<N>, ...>` の使用例がコンパイル・実行可能。
